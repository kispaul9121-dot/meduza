# Аудит Модели Данных

## Текущие сущности (Medusa Backend)

1. **ServerModel**
   - Представляет базовое шасси / вариант (например, DL360 Gen10 8SFF).
   - Характеристики заданы явно колонками: `brand`, `family`, `generation`, `chassis_type`, `drive_bays_front`, `cpu_socket`, `ram_slots_total` и т.д.
   - Минусы: Плохо масштабируется на новые типы оборудования (СХД, коммутаторы), так как колонки специфичны для серверов.

2. **Component**
   - Представляет конкретную деталь (CPU, RAM, Drive).
   - Часть базовой информации в колонках: `type`, `brand`, `model`, `price`.
   - Вся техническая специфика убрана в **JSON-поле `specs_json`**.
   - Минусы: Нет контроля типов, возможны опечатки, сложно индексировать и фильтровать на уровне БД.

3. **CompatibilityRule & RulePreset**
   - Гибкая система правил (через JSON `conditions_json` и `action_json`).
   - Отлично спроектирована, работает на бэкенде.

4. **Configuration**
   - Сохраненная конфигурация пользователя.
   - Данные лежат в JSON (snapshot). 

## Выявленные проблемы и дубли

1. **Слабая типизация характеристик**: В таблице `Component` всё техническое описание (ядра, частота, тип интерфейса) лежит в `specs_json`. Из-за этого мы видим в коде `specs.interface || specs.interfaces?.join?.(" ")`, что говорит о нестабильной структуре данных.
2. **Отсутствие справочников**: Нет таблиц для "Бренд", "Поколение", "Тип памяти". Они пишутся строками, что может привести к дублям ("Gen 10", "Gen10").
3. **Дублирование каталога**: Medusa имеет свои встроенные `Product` и `ProductVariant`. Кастомные `ServerModel` и `Component` существуют параллельно. В `ServerModel` есть поле `medusa_product_id`, что указывает на попытку связать кастомный каталог со стандартным, но сейчас это вызывает разрыв шаблона при добавлении в корзину.

## Целевая структура (План миграций)

Для долгосрочного масштабирования на СХД, сетевое оборудование и новые категории требуется внедрение паттерна EAV (Entity-Attribute-Value) или жестких схем.

1. **Категории и Атрибуты**:
   - `Category` (Серверы, Процессоры, Жесткие диски)
   - `AttributeDefinition` (Ядра, Тип памяти, Форм-фактор, Интерфейс) с типами (string, number, enum)
   - `AttributeValue` (Связь товара с атрибутом)

2. **Унификация с Medusa**:
   - Максимально использовать штатный `Product` для отображения в каталоге, а `ServerModel` / `Component` оставить как расширения для Rules Engine и конфигуратора.

## Безопасный порядок будущих миграций

1. **Не трогать текущую БД** (работает и выполняет задачи).
2. Создать новые таблицы `AttributeDefinition` и `AttributeValue`.
3. Написать скрипт миграции, который пройдется по `Component.specs_json` и аккуратно извлечет оттуда ключи, создав атрибуты.
4. Перевести фильтры в API на использование нормализованных атрибутов (вместо извлечения из JSON).
