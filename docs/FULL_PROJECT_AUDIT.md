# Полный Аудит Проекта Payloud на Medusa (12 этапов)

## 1. Executive Summary

Проект Payloud на данный момент представляет собой гибридную архитектуру, где мощный и продуманный backend (Rules Engine для конфигуратора на базе Medusa) соседствует с частично реализованным, прототипным frontend (Next.js).

Главное достижение проекта: **Конфигуратор серверов (Rules Engine) полностью рабочий, покрыт тестами, правильно валидирует зависимости и считает цену.**

Главные проблемы проекта:
1. **Каталог и фильтрация реализованы на клиенте**. Все серверы грузятся в браузер, фильтры применяются через `useMemo`. Это не отмасштабируется.
2. **Фиктивные маршруты**. Избранное, сравнение, корзина и даже каталог комплектующих реализованы через параметры URL (например, `?view=compare`). Это ломает SEO и пользовательский опыт.
3. **Разрозненность каталога Medusa и Кастомного**. Кастомные сущности `ServerModel` и `Component` частично оторваны от стандартных `Product` платформы Medusa.

Общая готовность проекта: **55%**

---

## 2. Архитектура и Структура

*   **Monorepo (Turborepo)**: `apps/backend` (Medusa), `apps/storefront` (Next.js App Router).
*   **Конфигуратор**: Кастомный модуль в Medusa (`modules/server-configurator`).
*   **Импорт/Seed**: Мощная система скриптов (`scripts/seed-configurator-defaults.ts`).
*   **Админка**: Кастомные маршруты админки (`admin/routes/server-configurator`).

## 3. Детальный аудит по этапам

### Этап 1. Карта сайта, маршруты и навигация
*   **Статус:** FUNCTIONAL
*   **Готовность:** 75%
*   **Что найдено:** Реализован `server-header.tsx` с Mega Menu. Есть роуты `/servers` и `/servers/[slug]`.
*   **Отсутствует/Проблемы:** Роуты комплектующих, избранного, сравнения работают через `?view=...` и `?component=...`. Отдельных страниц для них нет.
*   **Следующее действие:** Перенести фиктивные query-маршруты в полноценные страницы `app/`.

### Этап 2. Словарь данных и классификация
*   **Статус:** PARTIAL
*   **Готовность:** 60%
*   **Что найдено:** Сущности `ServerModel` и `Component`. Генерация фасетов для фильтров работает.
*   **Отсутствует/Проблемы:** Все характеристики засунуты в `specs_json`. Нет жесткой схемы и контроля типов. Возможен хаос при импорте.
*   **Следующее действие:** Нормализация атрибутов (создание справочников).

### Этап 3. Шаблоны страниц
*   **Статус:** FUNCTIONAL
*   **Готовность:** 80%
*   **Что найдено:** Качественная верстка конфигуратора, карточки товара.
*   **Отсутствует/Проблемы:** Жестко зашитые SEO-данные в `/servers/page.tsx` (`title: "HPE DL360 Gen10 servers"`). Нет шаблонов для решений.
*   **Следующее действие:** Интеграция CMS для динамических страниц.

### Этап 4. Backend-каталог, поиск, фильтры
*   **Статус:** PARTIAL (Блокер)
*   **Готовность:** 50%
*   **Что найдено:** Фасеты генерируются на бэке. Но сама фильтрация (`catalog-client.tsx:180`) происходит на клиенте: `models.filter((model) => matches(model, selected))`.
*   **Отсутствует/Проблемы:** Отсутствует серверная пагинация, серверный фильтр, серверный поиск.
*   **Критический риск:** При добавлении 1000 товаров браузер будет зависать.

### Этап 5. Страница модели и Конфигуратор
*   **Статус:** PRODUCTION_READY
*   **Готовность:** 90%
*   **Что найдено:** Конфигуратор работает отлично. Выбор процессора меняет доступную память, проверяются охлаждение, корзины дисков. Rules Engine на бекенде валидирует всё через `evaluateConditions`.
*   **Отсутствует:** Кнопка "Поделиться сборкой" (share link).

### Этап 6. Готовые конфигурации
*   **Статус:** EARLY_PROTOTYPE
*   **Готовность:** 30%
*   **Что найдено:** Модель `Configuration` существует в бекенде. Но она больше используется для сохранения корзины.
*   **Отсутствует:** Нет UI в каталоге, чтобы вывести "Сборка для 1С", "Сборка для Виртуализации".

### Этап 7. Каталог комплектующих
*   **Статус:** PARTIAL
*   **Готовность:** 45%
*   **Что найдено:** Компоненты есть в базе.
*   **Отсутствует:** Нет отдельной страницы для покупки комплектующих без сервера. Меню ссылается на `/servers?component=cpu`.

### Этап 8. Контент, Решения и База знаний
*   **Статус:** NOT_STARTED
*   **Готовность:** 10%
*   **Что найдено:** `HelpAnnotation` (поповеры с подсказками) работают.
*   **Отсутствует:** Статьи, гайды, решения.

### Этап 9. Импорт и наполнение
*   **Статус:** FUNCTIONAL
*   **Готовность:** 75%
*   **Что найдено:** Крутые скрипты импорта с поддержкой dry-run и отчетами. Наличие админ-интерфейса `import-review`.

### Этап 10. Продажи, корзина, КП
*   **Статус:** FUNCTIONAL
*   **Готовность:** 85%
*   **Что найдено:** Создан крутой Medusa Workflow `addConfiguredServerToCartWorkflow`. Заказ собирается правильно.

### Этап 11. SEO и Производительность
*   **Статус:** PARTIAL
*   **Готовность:** 60%
*   **Что найдено:** Настроен `sitemap.ts`.

### Этап 12. Расширение архитектуры
*   **Статус:** EARLY_PROTOTYPE
*   **Готовность:** 40%
*   **Проблема:** Таблица `ServerModel` имеет колонки `cpu_socket`, `ram_slots_total`. Это не подойдет для коммутаторов или СХД, где нет CPU Socket. Нужна модель EAV.

## 5. Итоги и Критические Блокеры

**P0-Блокеры (Что нельзя делать сейчас):**
1. **НЕ ИМПОРТИРОВАТЬ МАССОВО ТОВАРЫ.** Клиентская фильтрация (`models.filter` в React) умрет. Сначала нужно перенести фильтрацию в API.
2. **Не запускать рекламу (SEO).** Ссылки на категории работают через `?view=...`, они не будут нормально индексироваться.

**Быстрые улучшения (Quick Wins):**
- Заменить хардкод H1 на странице каталога на динамический из пропсов.
- Вынести `Compare` и `Favorites` в отдельные роуты.
