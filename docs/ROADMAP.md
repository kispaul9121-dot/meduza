# Дорожная карта развития Payloud (Roadmap)

## Уровни приоритетов
* **P0** — блокирует дальнейшую разработку.
* **P1** — необходимо до первой рабочей версии (MVP).
* **P2** — нужно до массового наполнения.
* **P3** — развитие после запуска.

---

### Ближайшие действия (P0-P1)

| ID | Название | Этап | Приоритет | Зависимости | Оценка |
|---|---|---|---|---|---|
| 1 | **Серверная фильтрация каталога** | 4. Каталог | P0 | Нет | L |
| Цель: | Перенести логику фильтрации из `useMemo` клиента на backend (создать API с поддержкой параметров, пагинации и подсчета фасетов). Иначе приложение рухнет при добавлении 1000+ серверов. |
| 2 | **Устранение фиктивных маршрутов** | 1. Маршруты | P1 | Нет | M |
| Цель: | Создать реальные страницы (Next.js App Router) для Избранного, Сравнения и Комплектующих, вместо использования `?view=...`. Это критично для SEO. |
| 3 | **Синхронизация корзин** | 10. Корзина | P1 | Нет | M |
| Цель: | Объединить стандартную `cart` от Medusa и кастомную `configured-cart`. В данный момент существуют две параллельные реализации, что вызовет баги при оформлении заказа. |

### Первая полноценная версия (MVP) (P1-P2)

| ID | Название | Этап | Приоритет | Зависимости | Оценка |
|---|---|---|---|---|---|
| 4 | **Нормализация справочников (Схемы фильтров)** | 2. Данные | P2 | ID 1 | XL |
| Цель: | Внедрить таблицы характеристик вместо свободного `specs_json`. Без этого массовый импорт создаст хаос в фильтрах (появятся "gen10", "Gen10", "G10"). |
| 5 | **Отдельные страницы комплектующих** | 7. Компоненты| P1 | ID 2 | M |
| Цель: | Реализовать роуты `/components/processors`, `/components/memory` и т.д., чтобы комплектующие можно было покупать отдельно от конфигуратора, и они индексировались. |
| 6 | **SEO динамика страниц** | 11. SEO | P1 | Нет | S |
| Цель: | Заменить жестко прописанный title "HPE DL360 Gen10 servers" на странице каталога `/servers/page.tsx` на динамический. Настроить Canonical для отсечения дублей фильтров. |
| 7 | **Готовые конфигурации** | 6. Сборки | P2 | Нет | M |
| Цель: | Вывести сущности готовых сборок (сейчас есть в БД) в UI каталога и на страницу модели. Реализовать кнопку "Купить" для них. |
| 8 | **Интеграция CMS** | 8. Контент | P2 | Нет | L |
| Цель: | Добавить страницы решений (Виртуализация, 1С) и Базу знаний. Использовать Payload CMS (замечена в воркспейсе пользователя) или встроенные возможности. |

### Дальнейшее масштабирование (P3)

| ID | Название | Этап | Приоритет | Зависимости | Оценка |
|---|---|---|---|---|---|
| 9 | **Интеграция Supplier API** | 12. Расширение | P3 | ID 4 | XL |
| Цель: | Создать архитектуру для автоматического обновления цен и остатков от поставщиков. Требуется разделение на "Supplier Product" и "Published Product". |
| 10 | **Личный кабинет B2B** | 10. Продажи | P3 | ID 3 | L |
| Цель: | Реализовать просмотр истории запрошенных КП, скачивание PDF конфигураций, управление реквизитами компаний. |
| 11 | **Универсализация ServerModel** | 12. Расширение | P3 | ID 4 | XL |
| Цель: | Подготовить модель данных для СХД и коммутаторов (сейчас колонки заточены только под серверы, например `cpu_socket`). |

### Критерии для старта массового импорта товаров:
НЕ начинать массовый импорт (тысячи товаров) до завершения задач **ID 1 (Серверная фильтрация)** и **ID 4 (Нормализация справочников)**. Иначе фронтенд зависнет от объема данных, а фильтры сломаются от неконсистентности.
